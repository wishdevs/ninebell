# 미지급금 법인카드 전표 결재 — 프로세스 정의 (voucher-card-payable)

> 회계전표 그룹. 외상매출금/외상매입금(전표조회승인 조회+결재)의 **공유 백본을 재사용**하되,
> 카드 고유 확장 3가지(선행 결제번호 수집·2번째 메뉴 탭·결제창 내 참조문서 선택)를 얹는다.
> 상태: **v1 — 프로브 완료(2026-07-21). 메커니즘 대부분 ✅(다중탭·결의서조회승인 폼·참조문서
> dialog 구조). ⚠ 미해소: "결제번호" 정체(참조문서 검색 0건) + 대상 범위 — 사용자 확인 대기.**
> 형제(공유 백본): [../voucher_receivable/PROCESS.md](../voucher_receivable/PROCESS.md).

## 개요

전표조회승인에서 **미결·전자결재저장 상태의 일반 전표**를 조회한 뒤, 각 건을 결재창까지 열어
**가상 상신(로그만)** 한다. 단 매출/매입과 달리, 결제창에서 상신 전에 **참조문서(=미지급금
원전표의 결제번호)를 선택**해야 한다. 그 결제번호는 **2번째 메뉴(결의서조회승인)** 에서 미리
수집해 둔다. 실제 상신은 하지 않는다.

- 입력 방식: 실행 전 폼(조회조건 대부분 고정, 회계일만 월). HITL 개입 없음. 전체 진행.
- **영속 없음**: F7/F6/실제 상신 없음. 결제창·참조문서 선택 후 상신하지 않고 닫는다
  (참조문서 선택+확인이 비영속인지 프로브로 확정 — 아래 ❓).

## ⚠ 비가역 / 절대 안전
- **실제 상신 절대 금지**(가상 상신 로그만). '보관'도 미클릭(정책 확인 대기).
- 참조문서 '확인'까지가 결제창 내부 조작 — **상신 직전 단계**. 확인이 비영속인지(상신 안 하면
  참조문서 연결이 남지 않는지) 프로브로 반드시 확정. 불확실하면 관찰만 하고 중단.
- ⚠ EAP draft 잔존 이슈(형제와 동일): 결제창 열기만으로 EAP 임시문서 생성 가능(전체 진행 시 N건).
- 테스트는 격리 계정(이트라이브2/1111)에서만.

## 확정된 업무 결정(사용자, 2026-07-21)
- **D1 전표유형**: **일반**(SYSDEF_CD=11 — voucher-payable 프로브 때 존재 확인). 나머지 조회조건은
  매출/매입과 동일(회계단위 나인벨·작성부서 전체·회계일 당월·작성자 공백·역분개 전체·전표상태
  미결·전자결재상태 저장). → **공유 set_query 를 docu_types=("일반",) 로 재사용**.
- **D2 회계일**: 당월 고정이되 **변수화(유연하게)** — 추후 변경 가능하도록 파라미터/상수로.
- **D3 결제 팝업**: 외상매출금과 **같은 EAP 전자결재 팝업**(거기에 참조문서 선택 단계가 더 있음)
  → 결제창 열기/렌더대기/닫기 공유 프리미티브 재사용.
- **D4 처리 범위**: **전체 진행**(max_rows=None 기본, 형제와 동일).
- **D5 선행 수집(카드 고유)**: 전표조회승인 결과에서 **결의서번호 전부 수집** → 2번째 메뉴
  결의서조회승인에서 번호별 조회 → **결제번호 전부 수집**.
- **D6 참조문서 선택(카드 고유)**: 결제창 하단 '참조문서 선택' → 필터 펼침 → 문서번호에 해당
  **결제번호** 입력 → 조회 → 뜬 항목 선택 → 참조문서 그리드 **아래(down) 버튼**으로 '선택된
  문서 목록'으로 이동 → 확인 → (가상)상신.

## 카드 고유 플로우 상세(사용자 구술 — ❓ 실측 대상)
### Phase A — 전표조회승인(공유 백본): 조회 + 결의서번호 수집
- login → 회계 → 전표조회승인(GLDDOC00700) → set_query(docu_types=일반) → run_query.
- 결과 마스터그리드에서 **결의서번호 컬럼** 수집. ❓ 결의서번호 컬럼명(DOCU_NO? 별도 컬럼?)·
  전표조회승인 행과 결의서번호의 관계.

### Phase B — 2번째 메뉴 탭: 결의서조회승인에서 결제번호 수집
- 총계정원장 > 전표관리 > **결의서조회승인** 메뉴 열기. ❓ menu_id/deeplink.
- ⚠ **다중 메뉴 탭**: 옴니솔은 메뉴를 하나 더 열면 상단 로고 오른쪽에 **탭**으로 뜨고 페이지가
  **캐시**됨(전표조회승인 탭 유지). ❓ 탭 DOM 구조·탭 전환 방법(클릭 셀렉터)·2번째 탭 활성 판정.
- 새 페이지 조회폼: 회계단위 나인벨(기본)·**결의부서 전체선택**·**결의자 공백(기본값 삭제)**·
  **회계일 당월**·필터펼침 → **결의서번호 입력**(수집한 번호 하나씩). ❓ 각 필드 셀렉터
  (전표조회승인과 유사 위젯 추정 — 재사용 가능성). 결의서번호 입력란 셀렉터.
- 조회 → 결과에서 **결제번호 수집**. 수집한 결의서번호 수만큼 반복. ❓ 결제번호 컬럼명·
  결의서번호↔결제번호 매핑(1:1?).

### Phase C — 첫 탭 복귀 + 결제 loop + 참조문서 선택
- 캐시된 전표조회승인 탭으로 전환.
- 각 행: 체크 → 결제 → EAP 팝업(공유). 팝업 안에서:
  - '참조문서 선택' 버튼(하단). ❓ 셀렉터/좌표.
  - 필터 펼침 → 문서번호 = 이 행에 해당하는 **결제번호** 입력 → 조회. ❓ 문서번호 입력란·조회버튼.
  - 하단 리스트에 뜬 항목 선택 → 참조문서 그리드 **아래(down) 버튼** → '선택된 문서 목록' 이동.
    ❓ 항목 선택·아래버튼·두 그리드 셀렉터.
  - 확인 → 가상 상신(로그) → 닫기(상신 미클릭). ❓ 확인 버튼·비영속 여부.
  - ⚠ **매핑**: 전표조회승인 행 → 결의서번호 → 결제번호 → 참조문서. ❓ 이 매핑을 어떻게 잇는지
    (행에서 결의서번호를 읽어 Phase B 수집분과 매칭).

## 재사용 vs 신규
- **재사용(공유)**: login/user_type/menu_nav 진입, set_query(docu_types=일반), run_query, 결제창
  열기·렌더대기·닫기, D7 정합성, 본창 로딩대기, 진행 노출, 공지팝업 닫기.
- **신규(카드 고유, 프로브 필요)**: 다중 메뉴 탭 nbkit 헬퍼, 결의서조회승인 조회폼·결제번호 수집,
  참조문서 선택 sub-flow, 행↔결의서번호↔결제번호 매핑.

## 설계(권장 — 공유 백본 + 훅)
- `build_voucher_card_graph()` = 공유 백본 + **pre-collect 노드**(Phase A 결의서번호 → Phase B
  결제번호 수집) + **on_popup 훅**(Phase C 참조문서 선택)을 loop_approvals 에 주입.
- loop_approvals 에 optional `on_popup(child, ctx)` 훅 파라미터 추가(매출/매입은 None).

## 사용자 확정(2026-07-21)
- **결제번호 = 결의서조회승인 결과의 `GWDOCU_NO`("결재번호", 예 "(주)나인벨-2026-12961")** ✅.
  그 값을 참조문서 dialog 문서번호에 입력→조회하면 참조문서 목록이 나온다(사용자 실무 확인).
  → 매핑: 전표조회승인 행 `ABDOCU_NO` → 결의서조회승인 `GWDOCU_NO` → 참조문서 문서번호.
- **대상 = 결의구분=카드(`ABDOCU_FG_CD`=52)만** ✅. → Phase B 를 **결의구분=카드로 일괄 조회**해
  `ABDOCU_NO`→`GWDOCU_NO` 맵을 한 번에 만드는 접근 채택(번호별 반복 대신). Phase A 결과 중
  카드(52)·`ABDOCU_NO` 있는 행만 결제 대상.
- **참조문서 검색 0건은 테스트 계정의 시스템 승인 이슈(데이터측)** — 사용자 확정 2026-07-21:
  "플로우 맞음. 참조문서가 조회되는 게 맞는데 현재 시스템 승인문제로 리스트에 안 나올 뿐.
  **나온다고 가정하고 진행**, 추후 손봄." → 코드는 GWDOCU_NO 검색→매치→선택→아래버튼→(확인)→
  가상 상신 로직을 그대로 구현하되, **0건(현재 테스트 상태)이면 우아하게 로그**("참조문서 미검색
  — 시스템 승인 대기") 후 가상 상신. 결재함 유력값 **기결문서(종결)**. 참조문서 dialog 결재함
  옵션 11종: 전체·상신문서·기결문서·기결문서(종결)·수신참조문서·후결함·전결함·시행문서·수신상신
  문서·시행확인문서·이전문서.
- ⚠ 확인(참조문서 attach) 비영속 미확정 — 상신과 함께 **게이트**(기본 미클릭, 로그만). 현재
  테스트론 참조문서 0건이라 도달 자체가 안 되므로 데이터-부재로 안전. 승인 이슈 해소 후 확인 검증.
