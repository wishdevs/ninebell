# 단계 8 — 운영 (셀렉터 드리프트·트레이스·버전관리)

자동화는 배포로 끝나지 않는다. 옴니솔은 리스킨/버전업 때 클래스·id·좌표·컬럼이 바뀌고, 그러면
헤드리스가 **조용히 실패**한다. 운영은 그 드리프트를 빨리 잡고 되돌리는 일이다.

## 셀렉터 드리프트

**단일 소스 원칙** — 취약한 것들은 이미 한 곳에 모여 있다. 드리프트 시 그 파일만 고친다:

| 종류 | 단일 소스 | 바뀌면 증상 |
|---|---|---|
| CSS 셀렉터 | `nbkit/omnisol/selectors.py` | 로그인폼/그리드/툴바버튼/모달/코드피커 미검출 |
| in-page JS | `nbkit/omnisol/js_lib.py` | rowcount·getJsonRows·menu-check·user-type bbox·collect 실패 |
| 메뉴 매핑 | `nbkit/omnisol/menu_schemas.py` | 딥링크/service_url/사용자유형 오류 → MenuError |
| 에이전트 고유 JS | `<pkg>/js.py` | 그 문서종류의 스냅샷/버튼좌표 실패 |

**감지 방법**: 라이브 사이클(단계 7)을 주기적으로 1회 돌린다. 실패하면 원인 분류(단계 4 표):
셀렉터 드리프트인지, 타이밍인지, 필드부재인지. DOM/`getColumns()` 덤프로 실제 값을 확인하고
단일 소스만 갱신 → 사이클 재실행으로 회귀 확인.

## 트레이스 / 아티팩트 저장

- 실패 프로브·사이클은 스크린샷(PNG)+JSON 덤프를 `backend/e2e/artifacts/` 에 남긴다.
- 사이클 리포트 JSON(`<name>_smoke_cycle.json`)에 PASS/N·avg run·잔존 전표·전표번호를 축적.
- 라이브 런의 진행 이벤트(events 큐: step/status/ms/error)는 러너가 흘리고 영속한다 —
  운영 중 실패는 이 이벤트 스트림 + 아티팩트로 사후분석.
- 디버깅이 필요하면 `record_video_dir`+`slow_mo` 로 스토리보드 영상.

## PROCESS.md 버전 관리

PROCESS.md 는 살아있는 문서다. 변경마다:

- **상태 줄 갱신**: `상태: **v1 — 라이브 프로브 완료, 10/10 PASS(2026-07-XX).**`
- **검증 로그 append**(삭제 아님) — 언제 무엇이 ❓→✅ 됐는지, 실패·원인·수정 이력.
- **형제 크로스링크** 유지(`> 형제:`), 공통 변경은 형제에도 반영.
- **RESOLUTIONS.md 상태 컬럼** 갱신(⚪ 예정 → 🟡 진행중 → 🟢 구축). delay_scale 등 운영 튜닝은
  `__init__.py` register_workflow 주석에 근거와 함께 기록(예: 168s→135s 하향 근거).

## 드리프트 대응 런북

1. 라이브 사이클 1회 → 실패 스텝·아티팩트 확보.
2. 원인 분류(단계 4). 셀렉터/JS/메뉴 중 어느 단일 소스인지 특정.
3. DOM/getColumns 덤프로 실제 값 확인 → 단일 소스 1파일만 수정.
4. 프로브로 그 스텝만 재확인 → 사이클 재실행(회귀).
5. PROCESS.md 검증 로그 + 상태 줄 갱신, 커밋.
